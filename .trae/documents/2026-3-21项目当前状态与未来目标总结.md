# 项目当前状态与未来目标总结

**文档版本:** 1.0
**创建日期:** 2026-03-21

## 1. 历史任务与已完成目标总结

通过分析项目至今的所有文档和开发活动，项目的演进过程可归纳为以下三个阶段：

### 1.1. 初期：核心功能修复与重构
此阶段主要集中在解决应用的核心可用性问题，确保基础流程的稳定。
- **登录流程修复**: 解决了教师和学生两种角色的登录白屏、循环跳转、空指针等严重 Bug。
- **数据库初始化修复**: 修正了 `Student` 表 `sid` 字段的唯一性约束，并完善了测试数据的生成逻辑。
- **个人资料页修复**: 解决了因对象未初始化导致的空指针异常，保证了页面的正常加载。

### 1.2. 中期：UI/UX 统一与功能增强
在核心功能稳定后，此阶段着重于改善用户体验和代码的模块化。
- **设置页重构**: 将主题切换、WebDAV、缓存管理等分散的功能统一整合到新的设置页 (Fragment) 中。
- **人脸识别功能增强**: 实现了从相册中选择并识别小尺寸人脸，并能成功导入为新学生的功能。

### 1.3. 近期：网络化与架构升级
此阶段是项目的重大转折，目标是使应用从单机模式转向客户端-服务器模式。
- **网络接口规范定义**: 制定了与后端 `omniattend-core` 对齐的 API 接口规范。
- **头像加载网络化**: 实现了完整的头像缓存与网络加载机制，解决了头像无法显示的问题。
- **班级列表网络化**: 搭建了 Repository 和 ViewModel 架构，将班级列表页面从读取本地数据升级为从网络接口获取，并回写到本地数据库。

---

## 2. 当前核心目标

综合所有进展，当前项目的核心目标是：

**将 `FaceCheck` 应用从一个纯本地、依赖构造数据的单机 App，全面升级为一个对接真实后端服务、具备网络同步和数据持久化能力的、符合现代 Android 架构的客户端应用。**

具体体现在：
1.  **数据来源的转变**: 将所有核心数据（班级、学生等）的权威来源从本地数据库转变为后端 API。
2.  **架构的现代化**: 全面采用 Repository、ViewModel、LiveData、Retrofit 等组件，构建数据驱动的健壮架构。

---

## 3. 未完成的目标 (Next Steps)

基于当前已搭建好的架构，后续需要完成以下关键任务以实现核心目标：

1.  **完善数据同步逻辑**:
    - **任务**: `ClassroomRepository` 中 `saveClassroomsToDb` 的实现目前是简单的“删除-插入”。需要升级为更智能的同步策略，例如基于数据版本或时间戳进行差异化更新，以避免覆盖用户在本地的未同步修改。
    - **扩展**: 将此同步逻辑扩展到学生、考勤记录等其他核心数据模型。

2.  **实现“写”操作的网络请求**:
    - **任务**: 完成 `ClassroomListActivity` 中“新增班级”按钮的 `TODO` 功能，实现调用后端 API 创建新班级。
    - **扩展**: 为所有“写”操作（如添加/修改学生、更新头像、提交考勤等）实现对应的 `POST`/`PUT` 网络请求，并将其集成到相应的 Repository 中。

3.  **处理身份认证与密钥管理**:
    - **任务**: 移除 `ClassroomRepository` 中的硬编码 API Key。
    - **任务**: 移除 `ClassroomListActivity` 中的硬编码 `teacherId`。
    - **方案**: 实现从用户登录后的会话（如 SharedPreferences）中安全地获取 `token` 或 `apiKey` 以及 `teacherId`，并将其通过请求头或参数附加到所有网络请求中。

4.  **人脸识别接口对齐**:
    - **任务**: 这是上一个任务遗留的。需要开始分析并实现对 `1class` 和普通 `class` 学生的区分处理。
    - **任务**: 对默认班级组进行人脸识别区分度测试，并根据结果制定处理策略（如通过本地 SQL 进行标记或排除）。

5.  **集成应用内代理以加速网络**:
    - **目标**: 解决 Cloudflare 后端服务（Workers, D1, R2）在中国大陆访问速度慢、稳定性差的问题。
    - **方案**: 在 Android 应用内集成一个嵌入式的代理内核（如 Sing-box 或 V2Ray）。
    - **实现**: 
        - 允许用户在设置中配置自定义的订阅链接和选择代理节点。
        - 修改应用的网络请求层（例如，Retrofit 的 OkHttpClient），使其所有发往 Cloudflare API 的流量都通过应用内启动的本地代理服务。
    - **预期效果**: 大幅提升 API 响应速度和稳定性，真正利用 Cloudflare 的边缘计算能力。
